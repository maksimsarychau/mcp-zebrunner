# v9.0.1 — Shared OAuth Flow Store

**Advanced Zebrunner MCP Server** patch release. Fixes OAuth login in multi-replica HTTP deployments (Kubernetes, Cloud Run with >1 instance).

Full changelog since **v8.2.0** is included below for teams upgrading across the 8.x → 9.x line.

---

## Highlights (v9.0.1)

### Fixed

- **OAuth login across Kubernetes replicas** — `selfauth` and Okta HTTP modes no longer keep pending auth, issued codes, or DCR clients only in process memory. Flow state persists under `dirname(TOKEN_STORE_PATH)/oauth-flow` as encrypted per-key files (same volume as `tokens.enc`), fixing `Invalid or expired state` when `/authorize` and POST `/login` hit different pods.
- Encrypted store files are written with restrictive permissions (`0600`).
- Path containment checks in the OAuth flow store (`FileOAuthFlowStore`).

### Changed

- Optional env **`OAUTH_FLOW_STORE_DIR`** overrides the default OAuth flow directory.
- **STDIO mode unchanged** — no OAuth flow store on the STDIO startup path.

### Ops notes

- All replicas must share the PVC mounted at **`TOKEN_STORE_PATH`**.
- Session affinity on `/mcp` is still recommended (MCP session IDs remain in-memory per pod).

## Since v8.2.0

### v9.0.0 — Dual MCP coexistence (major)

Run **Advanced Zebrunner MCP** side-by-side with Zebrunner's official hosted MCP (beta) at `https://{workspace}.zebrunner.com/api/mcp` without tool-name collisions.

| Before (8.2.0) | After (9.0.x) |
|----------------|---------------|
| `create_test_case` | `adv_create_test_case` |
| `list_test_runs` | `adv_list_test_runs` |
| … all tools | `adv_<name>` |

**Breaking:** Legacy short tool names are **not registered by default**. Migrate prompts and scripts to `adv_<name>`, or set:

```bash
ZEBRUNNER_REGISTER_LEGACY_ALIASES=true
```

**New in 9.0.0:**

- MCP resource **`zebrunner://mcp-routing`** and **`about_mcp_tools` mode `routing`** — decision table for official vs advanced MCP
- Docs: [OFFICIAL_MCP_PARITY.md](https://github.com/maksimsarychau/mcp-zebrunner/blob/master/docs/OFFICIAL_MCP_PARITY.md), [OFFICIAL_MCP_DIFF.json](https://github.com/maksimsarychau/mcp-zebrunner/blob/master/docs/OFFICIAL_MCP_DIFF.json), [API_COVERAGE.md](https://github.com/maksimsarychau/mcp-zebrunner/blob/master/docs/API_COVERAGE.md)
- README **Dual-MCP setup** section
- Every tool description prefixed with `[Advanced Zebrunner MCP]`

### v8.3.0 — Regression Results Analyzer

- New tool **`adv_regression_results_analyzer`** — milestone/build regression summary from TCM Public API
- Auto-detect previous milestone; per-run coverage indicators (✅ / ⚠️)
- New MCP prompt **`regression-summary`**

### Docker & instance config

- **`zebrunner-config.json`** override priority:
  1. `ZEBRUNNER_CONFIG_JSON` (inline JSON)
  2. Volume mount at `/config/zebrunner-config.json`
  3. `ZEBRUNNER_CONFIG_PATH`
  4. CWD (local dev)
- Default config shipped in Docker image at `/config/zebrunner-config.json`

## Upgrade checklist (8.2.0 → 9.0.1)

1. **Tool names** — Update prompts/scripts to `adv_<name>`, or enable `ZEBRUNNER_REGISTER_LEGACY_ALIASES=true` during migration.
2. **Dual MCP** — Connect both official and advanced servers; use `@zebrunner://mcp-routing` to pick the right tool.
3. **HTTP / K8s** — Set `TOKEN_STORE_PATH` + `TOKEN_STORE_KEY`; mount a **shared volume** across replicas.
4. **Docker config** — Mount custom config or pass `ZEBRUNNER_CONFIG_JSON` (see [`.env.example`](https://github.com/maksimsarychau/mcp-zebrunner/blob/master/.env.example)).
